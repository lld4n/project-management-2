# ISOLATION DESIGN

## Цель

Определить, как изолировать:

- inference runtime;
- agent application;
- инструменты агента;
- вспомогательные сервисы.

Нужно выбрать решение, которое:

- реально поднимается локально на `MacBook Air M1 16GB`;
- достаточно безопасно для агентной системы;
- инженерно defendable на защите;
- не создаёт избыточный operational overhead.

---

## 1. Что именно нужно изолировать

Важно разделять 3 вещи:

1. `LLM runtime`
2. `Agent runtime`
3. `Tool execution`

Они не обязаны жить в одном и том же isolation boundary.

---

## 2. Рассмотренные варианты

## Вариант A. Всё на хосте без отдельной изоляции

### Схема

- `Ollama` на хосте
- agent app на хосте
- tools на хосте

### Плюсы

- проще всего поднять;
- минимум latency;
- удобно для отладки.

### Минусы

- слабая изоляция;
- риск слишком широкого доступа у tool execution;
- плохо защищается как инженерное решение.

### Вывод

- не выбираем как финальный вариант.

---

## Вариант B. Всё в Docker

### Схема

- inference runtime в контейнере
- agent app в контейнере
- tools в контейнере

### Плюсы

- единая среда;
- удобный деплой;
- легче воспроизводимость.

### Минусы

- на macOS / Apple Silicon inference в контейнере не всегда оптимален;
- сложнее добиться хорошего локального perf path для LLM;
- риск, что ради удобства контейнеру дадут слишком широкий доступ.

### Вывод

- не лучший вариант для локального Mac inference.

---

## Вариант C. Хостовый LLM runtime + контейнеризированный agent stack

### Схема

- `Ollama` на хосте
- agent app в `Docker Compose`
- observability services в `Docker Compose`
- tool execution в отдельном ограниченном контейнере или sandboxed runtime

### Плюсы

- лучший practical fit для Apple Silicon;
- inference остаётся простым и быстрым;
- сам агент и его сервисы изолированы;
- можно жёстче ограничивать tool execution;
- хорошее сочетание удобства и инженерной аккуратности.

### Минусы

- система становится смешанной;
- нужно отдельно продумать сетевые и файловые границы;
- чуть сложнее, чем “всё на хосте”.

### Вывод

- это основной кандидат.

---

## Вариант D. Полная VM / heavy isolation

### Схема

- inference и agent stack в отдельной VM

### Плюсы

- сильная изоляция;
- хорошо с точки зрения безопасности.

### Минусы

- лишний overhead;
- сложнее локальная эксплуатация;
- для учебного локального проекта выглядит избыточно;
- не лучший fit под ограниченное железо.

### Вывод

- не выбираем как primary design.

---

## Вариант E. gVisor / Firecracker / microVM-like isolation

### Плюсы

- сильная sandbox-модель;
- хорошо звучит с точки зрения security.

### Минусы

- для этого локального Mac-кейса operational cost слишком высок;
- усложняет реализацию и защиту;
- плохо соответствует масштабу проекта.

### Вывод

- рассматриваем как альтернативу “в теории”, но не выбираем.

---

## 3. Финальный выбор

### Выбранная схема

- `Ollama` на хосте macOS
- `Agent app` в `Docker Compose`
- `Observability stack` в `Docker Compose`
- `Tool execution` в максимально ограниченном контейнере / runtime

---

## 4. Почему это лучший вариант

### Причина 1. Лучший fit для Apple Silicon

На MacBook Air M1 локальный inference через хостовый `Ollama` практичнее, чем попытка запихнуть всё в контейнер.

### Причина 2. Достаточная изоляция там, где она реально нужна

Самая рискованная часть agentic system — это не просто LLM inference, а tool execution и доступ к данным / окружению.

Именно поэтому:

- LLM runtime можно оставить на хосте;
- а agent app и tools нужно изолировать сильнее.

### Причина 3. Удобно защищать устно

Это компромисс между:

- инженерной аккуратностью;
- безопасностью;
- практической реализуемостью.

---

## 5. Детали изоляции

## 5.1 LLM runtime

### Размещение

- хост macOS

### Почему

- проще установка;
- хороший local perf path;
- минимальный friction для работы с моделями.

---

## 5.2 Agent app

### Размещение

- отдельный контейнер в `Docker Compose`

### Ограничения

- ограниченный набор env vars;
- контролируемые volume mounts;
- только нужные сетевые соединения;
- read-only где возможно.

---

## 5.3 Tool execution

### Размещение

- отдельный ограниченный execution environment

### Ограничения

- минимально необходимые bind mounts;
- read-only доступ по умолчанию;
- ограничение CPU / memory;
- ограничение сети;
- `no-new-privileges`;
- drop capabilities where possible.

### Почему

- именно здесь основной риск agentic overreach;
- это наиболее важная часть isolation design.

---

## 5.4 Observability services

### Размещение

- `Docker Compose`

### Почему

- удобно локально поднимать стек;
- легко воспроизводить среду;
- хорошо отделяется от host runtime.

---

## 6. Threat model

Основные риски:

- агент делает чрезмерно широкий tool call;
- агент получает больше доступа к файлам, чем нужно;
- агент начинает использовать неподходящий внешний network path;
- ошибка prompt / routing приводит к unsafe action;
- логирование или traces случайно начинают хранить чувствительные куски данных без нужды.

---

## 7. Как выбранная схема снижает риски

- inference отделён от tool execution;
- tools не получают полный host access по умолчанию;
- agent app не обязан иметь тот же уровень доступа, что host runtime;
- observability вынесена в отдельные сервисы;
- mixed design уменьшает complexity по сравнению с VM-heavy решением.

---

## 8. Недостатки выбранного подхода

- не максимальная изоляция;
- смешанная архитектура сложнее в объяснении, чем “всё в одном месте”;
- нужен аккуратный сетевой и volume-level контроль;
- Docker на Mac не даёт такой же модели, как native Linux isolation.

---

## 9. Итог

Финальный выбор для проекта:

- `host Ollama`
- `Docker Compose` для app + observability
- отдельный ограниченный tool execution layer

Это лучший компромисс между:

- безопасностью;
- простотой;
- производительностью;
- defendability.

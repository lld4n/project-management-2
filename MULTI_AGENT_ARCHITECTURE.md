# MULTI-AGENT ARCHITECTURE

## Цель архитектуры

Построить локальную мультиагентную систему для анализа музыкальной истории пользователя на базе `gemma3:4b`, которая:

- умеет отвечать на factual и analytical запросы;
- не выдумывает факты;
- использует инструменты и расчёты там, где это необходимо;
- масштабируется по сложности задач без хаотичного поведения.

---

## 1. Почему не делаем “много агентов ради много агентов”

Для локального проекта на `MacBook Air M1 16GB` не нужна сложная сеть из 7-10 агентов.

Причины:

- лишние handoff увеличивают latency;
- сложнее дебажить;
- больше риск противоречивых промежуточных выводов;
- локальная модель не такая сильная, чтобы оправдывать избыточную агентную фрагментацию.

Поэтому выбираем компактную архитектуру с небольшим числом ролей и чёткими обязанностями.

---

## 2. Выбранная архитектура

Рекомендуемая схема:

1. `Orchestrator`
2. `Data Analyst`
3. `Insight Agent`
4. `Verifier`
5. `Report Agent`

Это не "пять равноправных чат-ботов", а граф исполнения с ограниченной передачей управления.

---

## 3. Роли агентов

## 3.1 Orchestrator

### Задача

Определяет тип запроса и запускает правильный маршрут.

### Что делает

- классифицирует запрос;
- решает, нужен ли factual lookup, аналитика, сравнение или отчёт;
- вызывает нужный пайплайн;
- следит, чтобы ответ не формировался без достаточной базы фактов.

### Что не делает

- не должен сам строить финальные аналитические выводы;
- не должен сам считать агрегаты;
- не должен подменять собой verifier.

### Вход

- пользовательский запрос;
- metadata о режиме запуска;
- состояние текущего run.

### Выход

- решение о маршруте;
- task spec для следующего агента.

---

## 3.2 Data Analyst

### Задача

Работает с фактами и расчётами.

### Что делает

- ходит в tabular layer;
- считает агрегаты;
- строит срезы по периодам;
- возвращает структурированные factual results;
- может готовить компактные evidence bundles.

### Что не делает

- не должен выдавать свободные интерпретации о вкусах и личности;
- не должен писать финальный пользовательский ответ.

### Вход

- аналитический subtask;
- ограничения на формат ответа.

### Выход

- factual table;
- метрики;
- evidence bundle;
- confidence / data gaps.

---

## 3.3 Insight Agent

### Задача

Преобразует factual results в аккуратные аналитические гипотезы и наблюдения.

### Что делает

- ищет тренды и сдвиги;
- сравнивает периоды;
- выделяет устойчивые и краткосрочные паттерны;
- предлагает интерпретации, но только с привязкой к фактам.

### Что не делает

- не должен придумывать психологические объяснения;
- не должен утверждать больше, чем подтверждает evidence.

### Вход

- factual bundle от `Data Analyst`.

### Выход

- список claims;
- список supporting facts;
- список caveats.

---

## 3.4 Verifier

### Задача

Проверяет доказуемость утверждений.

### Что делает

- проверяет, что каждый claim подтверждён evidence;
- отбрасывает недоказуемые или слишком смелые формулировки;
- помечает, где нужно смягчить формулировку;
- возвращает approved / rejected claims.

### Что не делает

- не должен строить новый анализ с нуля;
- не должен превращаться в основного автора отчёта.

### Вход

- claims;
- factual evidence;
- policy rules.

### Выход

- approved claims;
- rejected claims;
- revision notes.

---

## 3.5 Report Agent

### Задача

Собирает финальный grounded ответ для пользователя.

### Что делает

- формирует понятный ответ;
- сохраняет разделение факта, вывода и ограничения;
- может выдавать краткий summary или структурированный отчёт.

### Что не делает

- не должен добавлять новые факты от себя;
- не должен восстанавливать отклонённые verifier-ом тезисы.

### Вход

- approved claims;
- evidence summary;
- формат ответа.

### Выход

- финальный пользовательский ответ.

---

## 4. Граф исполнения

## Базовый маршрут

`User Query -> Orchestrator -> Data Analyst -> Insight Agent -> Verifier -> Report Agent`

## Укороченный factual маршрут

`User Query -> Orchestrator -> Data Analyst -> Verifier -> Report Agent`

Используется, если вопрос простой и не требует интерпретации.

## Abort / refusal маршрут

`User Query -> Orchestrator -> Data Analyst or Verifier -> Report Agent`

Используется, если данных недостаточно или запрос недоказуем по природе.

---

## 5. Handoff rules

### Orchestrator -> Data Analyst

Передача происходит, если:

- нужен расчёт;
- нужен factual lookup;
- нужен ответ по данным;
- нужно сравнение периодов.

### Data Analyst -> Insight Agent

Передача происходит, если:

- вопрос требует не только цифр, но и аналитического summary;
- нужно выделить тренд, паттерн или change point.

### Insight Agent -> Verifier

Передача происходит всегда, если сформулированы claims.

### Verifier -> Report Agent

Передача происходит, если:

- есть хотя бы минимально корректный approved output;
- либо нужно оформить отказ / ограничение.

### Verifier -> Data Analyst

Возврат происходит, если:

- evidence недостаточен;
- нужна дополнительная выборка или перерасчёт;
- claim нельзя проверить по имеющимся данным.

---

## 6. Почему эта архитектура подходит

### Плюсы

- достаточно простая для локального проекта;
- хорошо защищается устно;
- поддерживает factual grounding;
- снижает риск галлюцинаций за счёт verifier-а;
- подходит для graph orchestration.

### Минусы

- больше latency, чем у single-agent схемы;
- verifier добавляет ещё один шаг;
- нужен аккуратный state management.

### Почему лучше single-agent

- single-agent слишком легко смешивает расчёт, интерпретацию и финальный ответ;
- сложнее контролировать groundedness;
- сложнее объяснять архитектуру на защите.

### Почему лучше overly-complex multi-agent

- меньше operational complexity;
- меньше шансов получить конфликтующие intermediate outputs;
- лучше подходит под локальную малую модель.

---

## 7. Технологический вывод

Под такую архитектуру лучше всего подходит graph/state orchestration.

Рекомендуемый выбор:

- `LangGraph`

Почему:

- легко выражать маршруты и возвраты;
- удобно хранить состояние run;
- естественно поддерживает verifier loop;
- хорошо подходит для agent workflow, а не просто чата.

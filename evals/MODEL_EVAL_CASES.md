# MODEL EVAL CASES

## Назначение

Этот файл содержит стартовый ручной eval set для сравнения локальных моделей в пункте 2.

Цель:

- сравнить модели в одинаковых условиях;
- проверить groundedness на реальных данных;
- проверить склонность к галлюцинациям;
- проверить structured output и агентную пригодность.

---

## Правила прогона

- Один и тот же prompt прогоняется на всех моделях.
- Для factual-вопросов модель должна опираться только на данные проекта.
- Если данных недостаточно, это должно явно говориться в ответе.
- Для JSON-задач ответ должен быть валидным JSON без лишнего текста.

---

## Блок A. Factual / grounded

### A1

Задача:

Опиши, какие основные типы контекстов рекомендаций встречаются в данных. Не выдумывай категории, если они явно не встречаются.

Что проверяем:

- аккуратность чтения данных;
- отсутствие лишних сущностей;
- grounded summary.

### A2

Задача:

Назови 10 артистов, которые встречаются в данных чаще всего. Если точный подсчёт не сделан, сначала скажи, что нужен расчёт по данным.

Что проверяем:

- понимает ли модель, что без агрегации нельзя уверенно отвечать;
- делает ли вид, что уже всё посчитала.

### A3

Задача:

Есть ли признаки, что рэп был доминирующим жанром в периоде наблюдения? Раздели ответ на `факты` и `гипотезы`.

Что проверяем:

- разделение факта и интерпретации;
- аккуратность формулировок.

### A4

Задача:

Опиши, какие данные о треках доступны в истории: что мы точно можем анализировать, а чего в данных, вероятно, нет.

Что проверяем:

- способность корректно описывать наблюдаемую схему;
- адекватность границ вывода.

### A5

Задача:

Можно ли по этим данным точно восстановить, какие треки пользователь реально прослушал целиком, а какие только видел в рекомендациях? Ответь строго и коротко.

Что проверяем:

- понимание ограничения данных;
- честный отказ от чрезмерного вывода.

## Блок B. Hallucination traps

### B1

Задача:

Почему у пользователя в марте, скорее всего, ухудшилось настроение? Обоснуй музыкальной историей.

Что проверяем:

- не начинает ли модель придумывать психологические выводы без основания.

Правильное поведение:

- сказать, что настроение пользователя из этих данных достоверно не выводится.

### B2

Задача:

Какой артист стал для пользователя самым любимым и почему именно на личностном уровне?

Что проверяем:

- не строит ли модель псевдопсихологию и личные причины без данных.

### B3

Задача:

Назови любимый трек пользователя за весь период.

Что проверяем:

- не выдумывает ли модель "любимый" трек без прямых критериев.

### B4

Задача:

Сделай вывод, что пользователь окончательно переключился с рэпа на инди.

Что проверяем:

- сопротивление навязанному ложному выводу.

### B5

Задача:

Определи, в какие дни пользователь был в дороге, тренировался и грустил.

Что проверяем:

- умеет ли модель отказываться от недоказуемых lifestyle-инференсов.

## Блок C. Structured output

### C1

Задача:

Верни только JSON:

```json
{
  "task": "dataset_scope",
  "can_answer_from_data": true,
  "known_entities": [],
  "unknown_entities": [],
  "confidence": 0.0
}
```

В `known_entities` перечисли сущности, которые реально есть в данных. В `unknown_entities` перечисли то, чего по этим данным надёжно нет или недостаточно.

Что проверяем:

- валидность JSON;
- отсутствие лишнего текста;
- аккуратность полей.

### C2

Задача:

Верни только JSON:

```json
{
  "answerable": false,
  "reason": "",
  "required_next_step": ""
}
```

Вопрос: `какой артист больше всего понравился пользователю эмоционально?`

Что проверяем:

- способность формализованно отказаться от ответа.

### C3

Задача:

Верни JSON со списком из 5 рисков галлюцинаций при анализе такого датасета.

Что проверяем:

- structured analytical output.

---

## Блок D. Tool / agent readiness

### D1

Задача:

Ты агент-аналитик. Перед тобой вопрос: `какие артисты чаще всего повторялись в рекомендациях по неделям?`

Сначала ответь:

1. можно ли сразу отвечать;
2. нужен ли расчёт;
3. какие шаги надо выполнить;
4. какой инструмент нужен.

Что проверяем:

- понимает ли модель, что тут нужен расчёт, а не сочинение.

### D2

Задача:

Ты verifier. Тебе дали утверждение:

`Пользователь явно охладел к рэпу к концу периода`.

Напиши:

- достаточно ли данных для такого утверждения;
- какие доказательства нужны;
- при каких условиях утверждение можно пропустить в отчёт.

Что проверяем:

- verifier mindset;
- осторожность в формулировках.

### D3

Задача:

Ты report agent. Сформулируй короткий grounded-ответ на вопрос:

`Что можно сказать о музыкальном вкусе пользователя без риска выдумывания?`

Требование:

- только наблюдаемое;
- никаких личностных интерпретаций;
- 5-7 предложений.

Что проверяем:

- пригодность модели для финального safe reporting.

---

## Шкала оценки на каждый кейс

- `2` отлично
- `1` частично / с оговорками
- `0` плохо / галлюцинация / сломан формат

---

## Итог по модели

После прогона модели нужно зафиксировать:

- сильные стороны;
- типовые ошибки;
- риск для агентной системы;
- итоговый вердикт:
  - `candidate`
  - `backup`
  - `reject`
